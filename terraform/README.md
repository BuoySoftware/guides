# Terraform

## Resource Naming

- Prefer lowercase, underscored resource names: `my_special_resource` not `my-special-resource`.
- Prefer singular resource names: `aws_s3_bucket.log` not `aws_s3_bucket.logs`.
- Use simple names (`main`, `this`, `default`) when a resource is the only one of its type.
- Use meaningful names when differentiating between multiple resources of the same type.
- Don't repeat the resource type in the name: `data "aws_iam_policy_document" "assume_role"` not `data "aws_iam_policy_document" "iam_policy_document"`.

## Context Module

- Use the [context module] for all resource naming and tagging.
- Pass context to child modules by merging with a `name` and optional `suffix`:
  `merge(module.context.context, { name = "eks" })` or
  `merge(module.context.context, { name = "rds", suffix = "buoyrails" })`.
- Use a separate platform context (`client = "buoy"`, `environment = "pltf"`)
  for shared resources that are not client-specific.

[context module]: https://github.com/BuoySoftware/terraform-infrastructure-aws/tree/main/modules/context

## Modules

- Place reusable modules in `modules/` of [terraform-infrastructure-aws].
- Ensure modules are account, client, and environment agnostic.
- Prefer input variables over hard coded values for easier reuse and extensibility.
- Ensure input variables have a type, description, and validation.
- Prefer a pinned, remote source path with a commit hash:
  `source = "git@github.com:BuoySoftware/terraform-infrastructure-aws.git//modules/eks/cluster?ref=d8b5f39"`
- Only use relative source paths in the `r-d` account for rapid iteration:
  `source = "../../modules/eks/cluster"`
- Ensure every module includes `main.tf`, `variables.tf`, `outputs.tf`, and a
  `README.md` generated by `bin/generate-docs.py`.
- See [example_module](example_module) for a minimal template.

[terraform-infrastructure-aws]: https://github.com/BuoySoftware/terraform-infrastructure-aws

## Directory Structure

- Use the `accounts/` directory of [terraform-infrastructure-aws] for all new
  infrastructure.
- Organize one directory per AWS account with a single Terraform state file.
- Organize one `.tf` file per resource type (e.g., `vpc.tf`, `eks.tf`, `s3.tf`).
- Use separate `kubernetes/` and `rds/` subdirectories with their own state
  files, as they require special connection methods.
- Don't create new resources in the legacy Terragrunt directories (`/prod`,
  `/non-prod`, `/r-d`, `/data-engineering` at the repo root). Those are
  maintained for existing resources only.

## State Management

- Use S3 for state storage with DynamoDB for locking.

## Provider Configuration

- Use standardized provider versions across all accounts.
- Apply `managed_by = "Terraform"` to all resources via `default_tags`.
- Configure a disaster recovery provider alias (`use2`) for us-east-2 in each
  account.

## Safety

- Always review `terraform plan` output before applying. Don't apply blindly.
- Don't use `-auto-approve`.
- Don't modify `.terraform.lock.hcl` manually — use `terraform init -upgrade`.
- Avoid large PRs that touch multiple unrelated subsystems.
- Flag any plan showing resource destruction or replacement to a human before
  proceeding.
- Include Terraform plan output in all PRs that modify infrastructure using
  the `<details>` / `<summary>` collapsible format.
