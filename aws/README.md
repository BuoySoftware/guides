# AWS

## Resource Naming

- Prefer lowercase, dashed resource names: `eks-cluster` not `eks_cluster`.
- Use names generated by the [context module] following the format
  `{client}-{environment}-{region}-{name}-{suffix}`, e.g., `jp-prod-use1-eks`
  or `jp-prod-use1-rds-buoyrails`.
- Use short client abbreviations: `jp`, `csl`, `buoy` (internal/shared).
- Use short environment abbreviations: `prod`, `non-prod`, `cicd`, `r-d`,
  `data-eng`, `pltf` (platform/shared).
- Use [fixed region abbreviations]: `use1` (us-east-1), `use2` (us-east-2, DR),
  `usw2` (us-west-2), `gbl` (global).

[context module]: https://github.com/BuoySoftware/terraform-infrastructure-aws/tree/main/modules/context
[fixed region abbreviations]: https://docs.cloudposse.com/github-actions/library/actions/aws-region-reduction-map/

## Tagging

- Ensure all Terraform-managed resources receive `managed_by = "Terraform"` via
  the AWS provider's `default_tags`.
- Use the [context module] to apply client, environment, and region tags
  automatically.

## Infrastructure Management

- Manage all infrastructure in [terraform-infrastructure-aws] under
  `accounts/<account-name>/`.
- Use separate AWS accounts to isolate workloads: `prod` for production,
  `non-prod` for dev/staging/validation/training, `cicd` for pipelines and
  ArgoCD, `data-engineering` for data platform, `r-d` for research.

[terraform-infrastructure-aws]: https://github.com/BuoySoftware/terraform-infrastructure-aws

## Network Architecture

- Use private subnets for EKS clusters. Access via SSM tunnel
  (`bin/connect-eks.py`).
- Use isolated database subnets for RDS clusters. Access via SOCKS5 proxy
  (`terraform-proxy.sh`).
- Use VPC peering for cross-account connectivity (e.g., CICD to prod/non-prod).

## Disaster Recovery

- Configure a secondary AWS provider for us-east-2 (`use2`) in each account.
- Use S3 cross-region replication for backup copies of critical buckets.
